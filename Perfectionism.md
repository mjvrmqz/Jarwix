# Jarwix Guide — Perfectionism Layer

## Overview

Perfectionism increases the perceived requirement for quality before action.

It raises the internal threshold required to start, continue, or finish tasks.

It affects execution across:
- Avoidance Loop
- Productive Procrastination Loop
- Execution Initiation Block

---

## 1. Core mechanism

Perfectionism operates as a threshold filter applied before action.

It increases required certainty in:
- readiness
- correctness
- completeness
- quality

before execution is allowed.

---

## 2. System effects

### Avoidance Loop
- increases ambiguity threshold
- delays entry action
- reinforces “not ready yet” state

---

### Productive Procrastination Loop
- increases optimization of non-critical systems
- shifts focus toward tools instead of output
- extends refinement cycles

---

### Execution Initiation Block
- increases perceived risk of imperfect action
- blocks initiation despite clarity
- delays first contact with task

---

## 3. Mechanism flow

### Step A — Evaluation inflation
Quality requirements increase above baseline task needs.

---

### Step B — Readiness delay
Action is postponed until conditions feel optimal.

---

### Step C — Substitution activity
Time is redirected to preparation, planning, or refinement.

---

### Step D — Execution deferral
Core task execution does not begin.

---

## 4. Core failure pattern

Perfectionism substitutes preparation for execution.

It increases perceived safety of waiting while reducing real progress.

---

## 5. Intervention rules

### STEP 1 — Identify threshold inflation
“I need this to be better before starting.”

---

### STEP 2 — Reset execution standard
Allow imperfect first version.

---

### STEP 3 — Enforce minimum viable execution
Perform smallest action that directly affects the real task output.

---

### STEP 4 — Separate execution from refinement
Optimization is only allowed after execution has begun.

---

## 6. Applies when

- delaying start due to quality concerns
- overworking planning or structure
- avoiding output due to imperfection risk
- refining systems instead of using them
- extended pre-execution preparation
